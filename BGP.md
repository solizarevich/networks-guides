### Виды сообщений BGP Message Types

 **Сообщение OPEN**

 Высылается после установления TCP сессии
  Содержит важную информацию:
  * Версия BGP
  * Номер AS
  * BGP RID
  * Hold Timer
  * Дополнительные параметры для согласования

 **KEEPALIVE**
   Подтверждает, что сосед жив и сессия активна.

  * Отправляется периодически
  * Значение Hold Timer определяет интервал
  * Если Keepalive не приходит вовремя — сессия считается упавшей.
  * По-умолчанию значение составляет 60 секунд
  * Значение «0» обозначает “не использовать KEEPALIVE”

  **UPDATE**
  Основной рабочий тип сообщения.
  Используется для:
  * объявления новых маршрутов (NLRI)
  * отзыва маршрутов (Withdrawn Routes)
  * передачи BGP атрибутов (AS-PATH, NEXT-HOP, MED, LOCAL PREF, Communities и др.)
  * для добавления или удаления префикса из BGP таблицы

 **ROUTE-REFRESH**
  Позволяет запросить у BGP соседа его (Adj-RIB-Out) полную пересылку маршрутов без пересоздания BGP-сессии.
  Используется при изменении политик (route-maps, фильтров и др.).

 **NOTIFICATION**
 
 Отправляется при ошибках или критических событиях.
 После отправки сессия немедленно закрывается.
 Используется для оповещения соседа об изменениях:
 
 * истечении Hold Timer
 * ошибках в сообщениях
 * изменении возможностей (Capabilities)
 * рестарт BGP сессии
 * истек Hold Timer
 * появились новые типы префиксов (Address Families)
 

### Механизмы предотвращения петель в BGP

BGP использует несколько механизмов, чтобы предотвратить петли маршрутизации:

 **1. Loop Prevention в eBGP**

Каждый маршрут, проходя через AS, получает запись в поле **AS-PATH**.
Если маршрутизатор видит свой номер AS в AS-PATH → маршрут отвергается → **петля невозможна**.

**Работает только в eBGP**, т.к. iBGP **не добавляет** свой AS в AS-PATH.

 **2. Split Horizon в iBGP**

Правило iBGP:

> **Маршрут, полученный через iBGP, нельзя передавать дальше другому iBGP-пиру.**

Почему так?

Потому что внутри одной AS:

* маршруты **не получают новый AS-PATH**
* значит, AS-PATH не может защитить от петель

Чтобы петли не появлялись между iBGP-роутерами, и существует правило split-horizon.


 **3. Из-за split-horizon нужно iBGP Full Mesh**

Если A, B и C — iBGP-роутеры:

* A получает маршрут от eBGP и может отправить его B и C
* но! B **не может** переслать его C
* C **не может** переслать его B

Чтобы все узлы получили маршруты, они должны быть **все-со-всеми** соединены iBGP-сессиями → **full mesh**.


###Как избежать full-mesh на практике?

В iBGP существует правило split horizon:

Маршрут, полученный через iBGP, нельзя переслать другому iBGP-соседу.

Из-за этого, если в сети есть N BGP-роутеров,
нужно сделать iBGP-сессии между каждым с каждым:

Формула:

N × (N – 1) / 2

Например:
10 роутеров → 45 iBGP-сессий
50 роутеров → 1225 сессий (!)

Это ужасно неудобно, тяжело администрировать и не масштабируется.
Route Reflector убирает необходимость полного меша.

✅ Route Reflector (RR) — самое популярное
Позволяет принимать маршруты от одного iBGP соседа и пересылать их другим iBGP клиентамнесмотря на правило split-horizon
То есть RR — это iBGP-“хаб”, через который все обмениваются маршрутами

ORIGINATOR-ID = Router-ID того клиента, который создал маршрут
CLUSTER-LIST — список всех RR, через которые прошёл маршрут


✅ Confederations

BGP Confederations нужны для решения той же проблемы, что и Route Reflector — плохая масштабируемость iBGP full-mesh, но решают её по-другому и с другими преимуществами.
Делят одну огромную AS на несколько маленьких под-AS. Уменьшают количество iBGP-пиров.
Каждая под-AS внутри себя использует обычный BGP (часто с RR), а между ними — особый режим BGP:
между под-AS работает как eBGP, но снаружи весь оператор выглядит как одна AS



### BGP Peer Group

Это группа соседей BGP, которым назначается общая конфигурация.
Вместо того чтобы повторять одинаковые настройки для каждого соседа, вы один раз задаёте их в группе — и подключаете туда соседей.

Пример: у вас 20 соседей, для всех нужны одинаковые route-map, обновления, фильтры.
С Peer Group вы пишете это один раз.

Зачем используются Peer Groups?

1. Упрощение конфигурации
Меньше повторяющихся строк — легче поддерживать и меньше ошибок.

2. Экономия CPU маршрутизатора

Исторически (до Cisco IOS 12.0 и до появления *peer templates* и *update-groups*) Peer Groups уменьшали количество расчётов при рассылке апдейтов.
Если соседи в Peer Group имеют одинаковые политики, то BGP **генерирует один update** и отправляет его всем участникам группы, вместо отдельного update для каждого соседа.

3. Легкость управления политиками

Хочешь изменить route-map для всех соседей группы?
Меняешь 1 строку — изменения применяются ко всем.

***Пример конфигурации (Cisco IOS)***

Создание Peer Group

```bash
router bgp 65000
  neighbor INTERNAL peer-group
  neighbor INTERNAL remote-as 65000
  neighbor INTERNAL update-source Loopback0
  neighbor INTERNAL next-hop-self
```

Добавление соседей в группу

```bash
  neighbor 10.0.0.2 peer-group INTERNAL
  neighbor 10.0.0.3 peer-group INTERNAL
  neighbor 10.0.0.4 peer-group INTERNAL
```

Добавление политик

```bash
  neighbor INTERNAL route-map RM-IN in
  neighbor INTERNAL route-map RM-OUT out
```

***Пример в JunOS***

```bash
protocols {
    bgp {
        group INTERNAL {
            type internal;
            local-address 192.0.2.1;
            neighbor 192.0.2.2;
            neighbor 192.0.2.3;
        }
    }
}
```

JunOS использует группы по умолчанию — они являются обязательной структурой.


### Выбор наилушего пути в Cisco - Best Path Selection

**При выборе маршрута BGP следует алгоритму:**

1. **Сравниваются только доступные маршруты до одного и того же префикса**

   * NEXT_HOP должен быть доступен (резолвиться в RIB)
   * AS_PATH не должен содержать номер собственной AS*

2. **Предпочитаются маршруты с большим Weight** (Cisco Proprietary) у Juniper - routing policy preference. У Huawei - apply cost 

3. **Предпочитаются маршруты с большим Local Preference**

4. **Предпочитаются маршруты с меньшей длиной AS_PATH**

5. **Предпочитаются маршруты с меньшим Origin (IGP < INCOMPLETE)**

6. **Предпочитаются маршруты с меньшим MED**

   * MED сравнивается только для маршрутов, полученных из одной и той же AS
   * Если MED отсутствует, полагают его значение наименьшим возможным*

7. **Предпочитаются маршруты, полученные по EBGP**

8. **Предпочитаются маршруты с меньшей IGP-метрикой до Next Hop**

9. **Предпочитаются маршруты, полученные от соседа с большим Router-ID**

10. **Предпочитается маршрут, полученный от соседа с большим IP-адресом**

#### BGP Weight — что это такое и зачем он нужен?

**Weight** — это *уникальный атрибут Cisco*, который существует **только локально на маршрутизаторе** и **никогда никуда не передаётся**.
Это самый первый критерий выбора маршрута, который BGP учитывает после проверки доступности next-hop.

То есть:
 **Если у маршрута Weight больше — он выиграет**, даже если все остальные атрибуты хуже.

---

Свойства Weight, которые важно помнить

 ✔ 1. **Локальный атрибут (не передаётся соседям)**

Weight виден только на этом конкретном роутере. Другие роутеры в AS его не увидят.

Это значит:

* На одном роутере маршрут может быть выбран как лучший (из-за Weight)
* На другом — не выбран (если Weight не настроен)

 ✔ 2. **Может быть назначен вручную**

Команды Cisco:

* глобально на соседа:

  ```
  neighbor x.x.x.x weight 500
  ```

* через route-map:

  ```
  set weight <value>
  ```

Можно направлять трафик очень гибко — например, заставить весь локальный трафик уходить через конкретного провайдера.

 ✔ 3. **Weight по умолчанию:**

* Маршруты, установленные вручную через `network`, `redistribute` → Weight = **32768**
* Все маршруты, полученные от соседей → Weight = **0**

То есть свои маршруты всегда важнее, чем полученные извне.

 ✔ 4. **Weight — абсолютный приоритет**

Если у двух маршрутов разный weight — BGP даже не смотрит Local Preference, AS_PATH, MED и т.д.

**Пример для понимания**

У тебя два провайдера:

* ISP1: Weight 200
* ISP2: Weight 0

Даже если ISP2 предлагает:

* короче AS_PATH
* лучше LocalPref
* меньше MED

Всё равно BGP выберет маршруты через **ISP1**, потому что Weight стоит выше всех остальных критериев.


**Где Weight реально используют?**

1. Управлять исходящим трафиком

Самый частый кейс:

Хочу, чтобы весь трафик уходил через провайдера №1.

Решение — поставить ему Weight 500:

```
router bgp 65000
 neighbor 1.1.1.1 weight 500
```

Готово.

2. “Жёсткая” приоритизация маршрутов

Если нужно *железно* выбрать конкретный путь, чтобы ничто его не перебило.

Weight гарантирует приоритет.

3. Тонкая настройка через route-map

Например: всё до 8.8.8.0/24 — через ISP1, всё остальное через ISP2.

```
route-map GOOGLE permit 10
 match ip address prefix-list GOOGLE
 set weight 600
```
